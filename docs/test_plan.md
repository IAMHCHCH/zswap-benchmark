# Zswap 性能测试计划

## 1. 测试目标

评估 **lz4 / lzo / zstd** 三种 zswap 压缩算法在 LLM 推理场景下的性能表现，重点分析：
- 线程数增加时性能带宽的线性变化
- 内存带宽饱和点
- 压缩比与 CPU 开销的平衡

## 2. 测试环境

### 硬件环境
- CPU: Intel/AMD 多核处理器
- 内存: 至少 16GB RAM
- 存储: SSD（用于模型文件）

### 软件环境
- Linux Kernel 5.x+ (启用 zswap)
- cgroup v1 或 v2
- llama.cpp (llama-bench)
- Python 3.8+ (用于结果分析)

## 3. 测试矩阵

### 3.1 算法对比

| 压缩算法 | 压缩比 (典型) | 压缩速度 | 内存开销 |
|----------|--------------|----------|----------|
| lz4 | ~1.8x | 最快 | 低 |
| lzo | ~2.0x | 中等 | 中 |
| zstd | ~2.8x | 较慢 | 高 |

### 3.2 线程数测试点

```
线程数: 1, 2, 4, 8, 12, 16, 32
```

### 3.3 内存限制测试点

```
内存限制: 2GB, 4GB, 8GB, 16GB
```

## 4. 测试场景

### 场景 1: 单请求推理（基线）
- 单个 LLM 请求
- 测试压缩算法基线性能
- 用于计算线性效率

### 场景 2: 并发推理（扩展性）
- 多线程并发处理请求
- 测试线程扩展性
- 检测内存带宽饱和点

### 场景 3: 批处理推理
- 批量请求处理
- 测试吞吐量和延迟平衡

## 5. 测试指标

### 5.1 性能指标
- **吞吐量**: tokens/s
- **延迟**: P50, P95, P99 (ms)
- **每秒请求数**: requests/s

### 5.2 资源指标
- **压缩比**: stored_pages / compressed_pages
- **CPU 使用率**: 压缩/解压缩 CPU 占用
- **内存带宽**: cache-misses, cache-references
- **上下文切换**: cs (context switches)

### 5.3 Zswap 特定指标
- pool_total_size: zswap 池大小
- stored_pages: 存储的页面数
- compressed_pages: 压缩后的页面数
- pool_limit_hit: 达到池限制的次数

## 6. 测试流程

```
1. 环境准备
   ├── 检查内核 zswap 支持
   ├── 安装依赖工具
   └── 编译 llama-bench

2. 基线测试 (无 zswap)
   └── 记录原始性能

3. 算法测试 (每种算法)
   ├── lz4 测试

   └── zstd 测试 (同上)

4. 数据分析
   ├── 计算线性效率
   ├── 找出饱和点
   └── 生成报告和图表
```

## 7. 线性度分析方法

### 理想情况
```
线程数翻倍 → 吞吐量翻倍
效率 = 100%
```

### 实际效率计算
```
效率_N = (吞吐量_N / 吞吐量_1) / N × 100%

例如:
- 1 线程: 100 tokens/s (基线)
- 2 线程: 180 tokens/s
  效率 = (180/100) / 2 × 100% = 90%
- 4 线程: 280 tokens/s
  效率 = (280/100) / 4 × 100% = 70%
```

### 饱和点判断
- 效率开始明显下降的线程数
- 通常发生在内存带宽饱和时

## 9. 注意事项

1. **数据备份**: 修改内核参数前备份重要数据
2. **隔离测试**: 使用 cgroup 隔离测试进程
3. **多次迭代**: 每组测试至少 3 次取平均值
4. **记录环境**: 记录所有系统配置以便复现
5. **监控资源**: 避免 OOM，使用 vm.panic_on_oom=0
